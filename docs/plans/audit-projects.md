# Projects Domain Audit

Deep function-by-function audit of `Ichor.Projects` domain and all 50+ modules.
Research-only. No files were modified.

---

## 1. DOMAIN MODULE: `projects.ex`

### Anti-pattern: 100% pass-through wrappers

Every public function in `projects.ex` is a zero-logic delegate to a resource
code_interface function. This is the canonical anti-pattern the `ash-elixir-expert.md`
warns about: the Domain should expose business capabilities, not wrap every CRUD
operation with an identical function signature.

**Current shape** (90 functions, all identical):
```elixir
def get_project(id), do: Project.get(id)
def create_project(attrs), do: Project.create(attrs)
def mark_loaded(project), do: Project.mark_loaded(project)
```

**What code_interface was designed to replace:**
The `code_interface do define(:get, ...) end` DSL in each resource already generates
`Project.get/1`, `Project.create/1`, etc. The Domain wrapper adds zero value and
doubles the call chain.

**The 4 non-trivial wrapper patterns in projects.ex:**

| Function | What it adds over the code_interface |
|---|---|
| `loaded_projects/0` | Swallows error, returns `[]` on failure |
| `all_projects/0` | Swallows error, returns `[]` on failure |
| `projects_by_status/1` | Swallows error, returns `[]` on failure |
| `advance_node/2` | Two-step: `get` then `advance` -- tiny orchestration |
| `load_node/2` | Two-step: `get` then `Ash.load` |

All other 85+ wrappers are pure noise.

**What domain functions SHOULD exist** (actual business capabilities):
- `spawn_mode_team/3` (currently in ModeSpawner -- orphaned from the domain)
- `spawn_dag_run/2` (currently in Spawner -- orphaned from the domain)
- `ingest_project_from_brief/1` (no domain entry point, driven by GenServer)
- `advance_node_to_next_stage/1` (the gate logic lives nowhere cleanly)

**Code_interface replacement table** -- every hand-written wrapper that is pure CRUD:

```elixir
# BEFORE (hand-written in projects.ex):
def get_project(id), do: Project.get(id)

# AFTER (already generated by code_interface, no wrapper needed):
Ichor.Projects.Project.get(id)
# OR called via domain with code_interface define on domain (Ash 3.x pattern)
```

## 2. ASH RESOURCES -- AUDIT

### 2.1 `Project` (project.ex)

**Well-formed.** Clean lifecycle via named update actions: `pick_up`, `mark_compiled`,
`mark_loaded`, `mark_failed`. Accept lists correct. Notifier attached.

**Observations:**
- `by_status` action takes `:atom` with inline `one_of` constraint -- could be
  promoted to use the existing `ProjectStatus` enum type for consistency, but not a bug.
- `list_all` is a read action wrapping `prepare(build(sort: ...))` -- idiomatic.
- No policies defined -- zero authorization checks on any action.

### 2.2 `Node` (node.ex)

**Well-formed.** The `advance` action takes a raw atom argument with inline
`one_of` constraint rather than using `NodeStatus` enum type -- minor inconsistency.

**Observations:**
- `by_project` argument typed as `:uuid_v7` but `Project` uses `uuid_primary_key(:id)`
  which is `:uuid`. This is a type mismatch risk.
- All child relationships (adrs, features, use_cases, etc.) are has_many but
  NOT loaded by default. The `load_node/2` Domain helper exists precisely because
  of this -- it's a band-aid over missing `preparations` that could pre-load associations.
- No policies.

### 2.3 `Adr` (adr.ex)

**Well-formed.** `status` uses inline atom enum rather than `Ash.Type.Enum`.

### 2.4 `Feature` (feature.ex)

**Well-formed.** Minimal. No `list_all` action (unlike ADR) -- inconsistency in the
design artifact sibling set.

### 2.5 `UseCase` (use_case.ex)

**Well-formed.** Same shape as Feature. Missing `list_all`.

### 2.6 `Checkpoint` (checkpoint.ex)

**Well-formed.** `mode` uses inline atom enum -- candidate for `Ash.Type.Enum`.

### 2.7 `Conversation` (conversation.ex)

**Well-formed.** `mode` uses inline atom enum -- same candidate.

### 2.8 `Phase` (phase.ex)

**Well-formed.** `status` uses inline atom enum (pending/in_progress/completed).
Has `sections` has_many but no code_interface for loading them.

### 2.9 `Section` (section.ex)

**Well-formed.** Has `tasks` has_many but no `list_all`.

### 2.10 `RoadmapTask` (roadmap_task.ex)

**Well-formed.** Uses `WorkStatus` enum (correct). Has subtasks has_many.

### 2.11 `Subtask` (subtask.ex)

**Well-formed.** Uses `WorkStatus`. No policies.

### 2.12 `Run` (run.ex)

**Well-formed.** Clean lifecycle actions. `by_node` and `by_path` both filter to
`status == :active` only -- this means archived runs are invisible to these queries,
which is intentional but worth noting.

### 2.13 `Job` (job.ex)

**Most complex resource.** Well-structured overall. Notable:

- `after_action` callbacks in `claim`, `complete`, `fail`, `reset` call
  `RuntimeCallbacks.after_job_transition/2` -- this is the correct Ash pattern
  for side-effect boundaries.
- `available` action uses a custom preparation `FilterAvailable` because SQLite
  cannot filter JSON arrays in SQL -- this is correct pragmatism.
- `claim` validates `attribute_equals(:status, :pending)` and
  `attribute_equals(:owner, nil)` -- idiomatic.
- `require_atomic?(false)` on mutating actions is appropriate since they use
  `after_action`.
- `now/0` is a `@doc false` public function used as `&__MODULE__.now/0` in
  `set_attribute` -- necessary Ash pattern, correctly documented.

## 3. PLAIN MODULES -- CLASSIFICATION

### 3.1 Pure Transformation (correct placement)

| Module | Shape | Notes |
|---|---|---|
| `Graph` | `[items] -> DAG structure` | Excellent pure module. Wave computation, critical path, file conflicts, pipeline stats. Zero DB calls. |
| `Validator` | `[items] -> validation results` | Pure cycle detection + ref checks. |
| `WorkerGroups` | `[jobs] -> [worker_groups]` | Pure file-overlap grouping. |
| `DagGenerator` | `(node_id) -> [task_maps]` | Mixed: DB call to load hierarchy, then pure transformation. The DB load could be extracted. |
| `SubsystemScaffold.Templates` | Template strings in -> String out | Perfectly pure. |

### 3.2 Orchestration (correct placement)

| Module | Shape | Notes |
|---|---|---|
| `Spawner` | `(node_id, project_id) -> {:ok, %{session, run}}` | Correct orchestrator. Loads data, computes worker groups, spawns agents. |
| `ModeSpawner` | `(mode, project_id, genesis_node_id) -> {:ok, session}` | Correct orchestrator for genesis mode teams. |
| `TeamLifecycle` | Facade over launch/cleanup modules | Thin delegation layer, correct. |
| `Loader` | `(tasks_jsonl_path | node_id, opts) -> {:ok, Run.t()}` | Two entry points with same output shape -- correct. |
| `DagGenerator` | `node_id -> {:ok, [task_maps]}` | Mixed-concern (DB + transformation) but acceptable. |

### 3.3 GenServer Lifecycle Processes (necessary concurrency)

| Module | Registry Key | What it monitors |
|---|---|---|
| `BuildRunner` | `{:run, run_id}` | MES manufacturing run -- tmux liveness, deadline, quality gates |
| `PlanRunner` | `{:genesis_run, run_id}` | Genesis mode run -- tmux liveness, coordinator delivery |
| `RunProcess` | `{:dag_run, run_id}` | DAG execution run -- tmux liveness, stale jobs, health checks |

**Key finding: IDENTICAL LIFECYCLE SHAPES across three GenServers.**

All three follow this exact pattern:
1. `start_link/1` registers via `{:via, Registry, {Ichor.Registry, {key, run_id}}}`
2. `via/1` helper for Registry lookup
3. `lookup/1` helper returning pid or nil
4. `list_all/0` using `Registry.select`
5. `init/1` subscribes to a signal category, schedules periodic checks
6. `handle_info(:check_liveness, state)` -- polls `Tmux.available?/1`, stops on false
7. `handle_info(%Message{...operator delivery...}, state)` -- completion detection
8. `cleanup(state)` -- calls `ModeRunner.kill_session/3` + `FleetSupervisor.disband_team/1`
9. `terminate/2` -- emits termination signal

**The only differences:**
- Registry key atom (`:run`, `:genesis_run`, `:dag_run`)
- Session prefix (`mes-`, `genesis-a/b/c-`, `dag-`)
- `BuildRunner` has `deadline_passed` state + corrective agent logic
- `RunProcess` has stale job detection + health check polling + `sync_job` cast
- `PlanRunner` matches on `String.starts_with?(from, state.session)` vs
  `RunProcess` matching `from == "#{session}-coordinator"`

### 3.4 Supervisor Modules -- IDENTICAL SHAPES

Three supervisor modules are structurally identical:

**`PlanSupervisor`**:
```elixir
use Supervisor
init(_opts) -> [{DynamicSupervisor, name: Ichor.Projects.PlanRunSupervisor, strategy: :one_for_one}]
Supervisor.init(children, strategy: :one_for_one)
```

**`ExecutionSupervisor`**:
```elixir
use Supervisor
init(_opts) -> [{DynamicSupervisor, name: Ichor.Projects.DynRunSupervisor, strategy: :one_for_one}]
Supervisor.init(children, strategy: :one_for_one)
```

**`LifecycleSupervisor`**:
```elixir
use Supervisor
init(_opts) -> more children (Janitor, ProjectIngestor, ResearchIngestor, CompletionHandler, Scheduler)
Supervisor.init(children, strategy: :rest_for_one)
```

`PlanSupervisor` and `ExecutionSupervisor` are both single-child wrappers around a
DynamicSupervisor. They provide no value over starting the DynamicSupervisor directly.

**`RunSupervisor`** is NOT a Supervisor -- it is a facade module (plain module with
`start_run/1`, `stop_run/1`, `list_active/1`). The naming is misleading.

### 3.5 Prompt Builder Modules -- DUPLICATED PATTERN

Three pure prompt-building modules:

| Module | Input | Output |
|---|---|---|
| `TeamPrompts` | `(run_id, roster)` | Strings for MES agent roles |
| `ModePrompts` | `(run_id, roster, node_id, brief)` | Strings for Genesis mode agents |
| `DagPrompts` | `(map())` | Strings for DAG execution agents |

**Structural observation:**
- `TeamPrompts` and `ModePrompts` both build role-specific prompts with `run_id` + `roster`.
- `DagPrompts` uses a map input (more flexible) but the same output contract.
- All three contain private formatting helpers (`format_wave_summary`, `format_steps`, etc.)
  that do the same thing (format lists).

**Duplicated private helpers:**
- `format_inline_list` in `DagPrompts` vs manual `Enum.join` in others
- `roster` building logic appears in both `ModeSpawner` (as `team_roster/2`) and
  `TeamPrompts` (as `roster/1`) -- slightly different but same shape
- `load_project_brief` in `ModeSpawner` formats project fields into a text block;
  similar pattern would be useful for all three prompt sets

### 3.6 TeamSpecBuilder (Projects) vs TeamSpecBuilder (Control)

**`Ichor.Projects.TeamSpecBuilder`** (this file):
- Input: `(run_id, team_name)`
- Delegates to `Ichor.Control.TeamSpecBuilder.build_from_state/2`
- Provides MES-specific prompt dispatch via `prompt_for_agent/3`
- Builds corrective team spec directly

**`Ichor.Control.TeamSpecBuilder`** (in control domain):
- The generic builder that accepts a `BlueprintState` and builder functions
- Produces `TeamSpec.t()` structs

**Assessment:** The split is correct. `Projects.TeamSpecBuilder` is a thin adapter
that bridges MES domain knowledge (roles, prompts, run_id) to the generic
`Control.TeamSpecBuilder`. No duplication -- appropriate layer separation.

## 4. SIGNAL / CALLBACK MODULES

### 4.1 `RuntimeSignals` -- Thin Signal Facade

Pure signal emission module. All functions are one-liners calling `Signals.emit/2`.
**Assessment:** Correct pattern. Centralizes signal names so callers don't hard-code atoms.

### 4.2 `RuntimeCallbacks` -- Ash After-Action Boundary

Called from `Job` resource after_action hooks. Emits signal + syncs run process.
**Assessment:** Correct Ash pattern. The `rescue` clause is defensive -- acceptable.

### 4.3 `ProjectIngestor` (GenServer)

Subscribes to `:messages` signal. Parses agent output (JSON or plain-text key-value)
into project attrs, calls `Projects.create_project/1`.

**Assessment:** Correct pattern (subscribe-then-ingest GenServer). Complex parsing
logic (`parse_key_value_brief`, continuation lines, list field conversion) is
embedded in the GenServer module. This parsing logic is pure transformation and
could be extracted to a separate module for testability.

### 4.4 `CompletionHandler` (GenServer)

Subscribes to `:dag` signal. On `:dag_run_completed`, resolves the node and project,
calls `SubsystemLoader.compile_and_load/1`, then calls `Projects.mark_loaded/1` or
`Projects.mark_failed/2`.

**Assessment:** Correct pattern. The resolution chain (`resolve_node` -> `resolve_project`)
uses a nil-guard pattern rather than Ash `with_filter` -- acceptable but verbose.
The `Projects.get_node({:ok, nil})` case is interesting: `get_by: [:id]` returns
`nil` on not-found in Ash, but the code guards against this explicitly.

## 5. OPERATIONAL MODULES

### 5.1 `Scheduler`

Ticks every 60s. Checks active `BuildRunner` count via Registry. Spawns one new
`BuildRunner` per tick if under `@max_concurrent`.

**Code smell:** Active run counting queries each pid via `GenServer.call/3` to
check `:deadline_passed?`. This is O(n) GenServer calls per tick. Duplicated
in both `handle_info(:tick)` and `handle_call(:status)`.

**Assessment:** Functional. The duplication between `:tick` handler and `:status`
call is a maintenance concern.

### 5.2 `Janitor`

Monitors `BuildRunner` pids via `Process.monitor`. Sweeps orphaned teams every 2 min.

**Assessment:** Correct OTP pattern. `rebuild_monitors/0` on init correctly
re-subscribes to existing processes after Janitor restart.

### 5.3 `Runtime` (GenServer)

This is the DAG **filesystem pipeline** runtime -- watches `tasks.jsonl` files,
NOT the Ash-backed DAG execution. Polls every 3s for project changes. Emits
`:dag_status` broadcast on every change.

**Key issue:** The name `Runtime` is ambiguous. This module manages the
**filesystem DAG** (tasks.jsonl based) while `RunProcess` manages the
**Ash DAG** (database backed). Two separate runtimes with similar names.

**Assessment:** Functional. The state map structure (watched_projects, pipeline,
dag, stale_tasks) is a plain map -- not an Ash resource. This is appropriate since
it's a transient view derived from filesystem state.

### 5.4 `Catalog`

State management helpers for `Runtime`. Maps project keys to filesystem paths.
Discovers projects from `~/.claude/teams/` and EventBuffer.

**Assessment:** Pure functions on state maps. Correct placement as support module.

### 5.5 `Discovery`

Scans filesystem paths for `tasks.jsonl` files. Reads `config.json` from team dirs.

**Assessment:** Pure discovery functions. No Ash involvement. Correct.

### 5.6 `DagAnalysis`

Parses `tasks.jsonl` files into normalized task maps, derives pipeline stats,
stale tasks, file conflicts via `Graph`.

**Assessment:** Mixed concern -- JSON parsing + DAG graph computation delegated to
`Graph`. The `normalize_task/1` function uses string-keyed map access rather than
the structured `Graph.to_graph_node/1` shape. Both exist in parallel.

### 5.7 `Actions`

Task mutation for the filesystem DAG (calls `JsonlStore`). Routes mutations through
`Catalog.tasks_jsonl_path_for_task/2`.

**Assessment:** Thin delegation. Correct but `parse_timestamp/1` is duplicated
almost exactly in `Graph.parse_timestamp/1`. Same ISO8601 parsing logic appears
in two places.

## 6. CROSS-CUTTING FINDINGS

### Finding 1: Domain wrappers are 95% noise

`projects.ex` has ~90 functions. At most 5 add business logic. The rest are
identity delegates. Callers already import the resource's code_interface.
The domain layer should define entry points for multi-step business operations,
not wrap every CRUD function.

**What to keep as domain functions:**
- `loaded_projects/0` and `all_projects/0` (swallow-error pattern for prompts)
- `advance_node/2` (two-step get + advance)
- `load_node/2` (two-step get + Ash.load)
- `projects_by_status/1` (swallow-error variant)
- The DAG entry points: `active_runs/0`, `runs_by_node/1`, `runs_by_path/1` (used for LiveView)

**What to remove from domain and call directly:**
- All `get_*/1`, `create_*/1`, `update_*/1`, `list_*/0` that are pure delegates
- These should be called as `Project.get(id)`, `Adr.create(attrs)`, etc.

### Finding 2: Three GenServer processes share identical lifecycle shape

`BuildRunner`, `PlanRunner`, and `RunProcess` have identical:
- Registry registration pattern (via tuple)
- `via/1`, `lookup/1`, `list_all/0` helpers
- Liveness polling loop
- Operator delivery detection
- cleanup/terminate pattern

This is ~60 lines of structural boilerplate per module (180 total) that could be
extracted to a shared `RunnerBase` behaviour or helper module. The domain-specific
logic (deadline/corrective agents in BuildRunner, stale jobs in RunProcess) is
~30-40 lines per module.

**Risk if extracted:** The session prefix, Registry key atom, and signal names
differ per runner. A `RunnerBase` would need to parameterize these 3 things.
Could use module attributes or callbacks.

### Finding 3: Two supervisor modules are trivial wrappers

`PlanSupervisor` and `ExecutionSupervisor` each wrap a single DynamicSupervisor
child. They add zero supervision logic. Both could be eliminated -- the
DynamicSupervisor could be started directly under `Application` or under
`LifecycleSupervisor` which already manages the MES DynamicSupervisor directly.

**`LifecycleSupervisor` already does this correctly:**
```elixir
children = [
  {DynamicSupervisor, name: Ichor.Projects.BuildRunSupervisor, strategy: :one_for_one},
  # + Janitor, ProjectIngestor, etc.
]
```

The genesis and DAG variants just wrap a single DynamicSupervisor with no siblings.

### Finding 4: `parse_timestamp/1` is duplicated

Appears in `Graph` (line 209) and `Actions` (line 80) with near-identical logic.
Both handle ISO8601 string -> DateTime, with the same `Z` stripping workaround
and NaiveDateTime fallback. Should be one function in one place.

### Finding 5: `RunSupervisor` is misnamed

`RunSupervisor` is a plain module (not a Supervisor) that wraps
`DynamicSupervisor.start_child` and `Registry.lookup`. It's a facade, not a
supervisor. Misleading name.

### Finding 6: `Runtime` vs `RunProcess` naming confusion

Two modules manage different "DAGs":
- `Runtime` = filesystem DAG (tasks.jsonl polling, state broadcast)
- `RunProcess` = Ash-backed DAG (Postgres jobs, tmux lifecycle)

`Status` is a thin delegate over `Runtime`. This name is also ambiguous.

### Finding 7: `ModeSpawner.load_project_brief/1` belongs in a view helper

`load_project_brief/1` formats a `Project` record into a multi-line string for
agent prompts. It's called from both `ModeSpawner` and `Spawner`. This is a
pure transformation from `Project.t()` -> `String.t()`. It lives in
`ModeSpawner` but is called by `Spawner` via `ModeSpawner.load_project_brief/1`.
The function should live in `TeamPrompts` or a `ProjectPresenter` module.

### Finding 8: `ProjectIngestor` parsing logic is not testable in isolation

`parse_key_value_brief/1` and its helpers (`parse_key_line`, `clean_value`,
`convert_list_fields`, `maybe_append_continuation`) are all `defp` inside a
GenServer. They're 80 lines of pure transformation that should be in a separate
`BriefParser` module or similar, enabling unit testing without spinning up the
GenServer.

### Finding 9: No authorization policies on any resource

Every resource has `defaults([:read, :destroy])` or explicit actions, but zero
`policies do ... end` blocks. All actions are effectively public within the Ash
domain. This is consistent (none of them have authorization) but worth noting
as a gap if the system ever requires actor-based access control.

### Finding 10: `ResearchContext` queries Domain at call time (no cache)

`ResearchContext.existing_subsystems/0` calls `Projects.loaded_projects()` on
every invocation. Since this is called during prompt generation (once per MES
run), and `loaded_projects()` does a DB query, this is acceptable. No cache needed.
But if called in a hot path, it would be a problem.

## 8. MODULES THAT COULD BECOME ASH RESOURCES

None of the plain modules in this domain are good candidates for becoming Ash
resources. The filesystem-backed modules (`Catalog`, `Discovery`, `DagAnalysis`)
deal with transient view state -- Ash resources require persistent identity.

`ResearchStore` wraps the external Memories HTTP API -- could be expressed as an
Ash resource with `AshJsonApiClient` but no clear benefit for this read-only usage.

- `Project`: all lifecycle transitions exposed

- `Phase`: no `list_all` -- minor inconsistency with siblings

## 11. PRIORITY REFACTOR CANDIDATES

**P1 -- High value, low risk:**
1. Collapse `projects.ex` domain: remove ~85 pass-through wrappers. Keep only the
   5 that add business logic. Callers use `Project.get(id)` etc. directly.
2. Fix `RunSupervisor` name -- rename to `RunFacade` or `DagRunRegistry`.
