# Phased Simplification Plan

This document turns the findings from [`2026-03-20-0916-audit-simplify.md`](./2026-03-20-0916-audit-simplify.md) into an execution plan.

The objective is not to rename modules or shuffle files. The objective is to remove duplicated control flow, collapse naming-only layers, and leave the system with a smaller number of state-owning runtimes.

Each phase should be completed in order. Do not start a later phase until the earlier phase has landed and the system is green. The phases are arranged so that low-risk structural cleanup happens first, shared runtime ownership is clarified second, and high-volume consolidation happens only after the core boundaries are stable.

## Phase 1: Delete Trivial Wrappers

### Purpose

Remove modules that add names but do not own meaningful state, behavior, or policy. This phase should improve clarity immediately while keeping behavioral risk low.

This phase is intentionally conservative. It should not change major runtime architecture. It should only remove indirection that already has an obvious owner elsewhere.

### Primary Targets

- `Ichor.Control.Lifecycle`
- `Ichor.Control.Lookup`
- `Ichor.Control.RuntimeQuery`
- `Ichor.Control.RuntimeView`
- `Ichor.Projects.PlanSupervisor`
- `Ichor.Projects.ExecutionSupervisor`
- `Ichor.Projects.RunSupervisor`
- `Ichor.Projects.RunnerRegistry`
- `Ichor.Projects.TeamLifecycle`
- `Ichor.Projects.TeamCleanup`
- `Ichor.Tools.MapUtils`
- `Ichor.Tools.AgentControl`
- any other pure delegation module found during implementation

### Instructions

1. Inventory every candidate wrapper before changing code.
   - For each target module, list:
   - who calls it
   - what module actually performs the work
   - whether it owns state, validation, or error mapping
   - whether it exists only to provide an alternate name
   - Do not delete a wrapper until the true owner is explicit.

2. Inline delegation into the real owner.
   - Replace `defdelegate`-style forwarding with direct calls from callers to the underlying module.
   - If multiple wrappers point to the same owner, standardize on one direct API and update all call sites.
   - If the wrapper adds argument normalization, move that normalization into the owner or the immediate caller. Do not preserve a separate helper layer just to keep the old shape.

3. Remove single-child supervisor wrappers.
   - Replace `PlanSupervisor` and `ExecutionSupervisor` with direct `DynamicSupervisor` children in the owning application or project supervision tree.
   - Replace `RunSupervisor` helpers with direct `DynamicSupervisor.start_child/2` and direct registry lookups where needed.
   - If named supervisors are still required, keep the names on the actual `DynamicSupervisor` children rather than preserving wrapper modules.

4. Remove registry convenience modules that only rename lookups.
   - Fold `RunnerRegistry` behavior into the module that starts and locates runner processes.
   - Keep registry naming conventions documented in one place, preferably next to the unified runner implementation that Phase 3 will introduce.

5. Collapse dashboard/query formatting helpers.
   - Move presentation-only shaping near the boundary that uses it.
   - If `RuntimeQuery` and `RuntimeView` only exist for dashboard serialization, combine them into one query/presenter module or inline them into the actual dashboard boundary.
   - Do not keep both a query module and a view module if they are formatting the same data.

6. Remove tool-only helper modules with no independent value.
   - Inline `MapUtils` into the small number of tool/resource modules that still need the transforms.
   - Merge `AgentControl` behavior into the tool implementation that actually issues the command.
   - Prefer private functions over shared utility modules unless at least three active modules use the helper and the helper expresses a real concept.

7. Delete the old modules only after call sites are fully updated.
   - Ensure no remaining compile-time references exist.
   - Remove now-unused aliases, tests, and docs references.

### Constraints

- Do not change public behavior unless the existing behavior is clearly accidental.
- Do not redesign the event system, runner lifecycle, or tool grouping in this phase.
- Do not add new wrappers to replace the deleted wrappers.

### Deliverables

- The listed wrapper modules are removed or merged.
- Call paths become shorter and easier to trace from caller to state owner.
- The supervision tree names still work without dedicated wrapper modules.

### Validation

- Run focused tests for the touched control, project, and tool modules.
- Run `mix compile`.
- Run `mix precommit` before closing the phase.

### Exit Criteria

- No deleted module is still referenced anywhere in `lib/` or `test/`.
- Every remaining wrapper can justify its existence by owning state, policy, or a boundary that should remain stable.
- The codebase is simpler to navigate without any change to runtime topology.

## Phase 2: Unify Messaging and Event Flow

### Purpose

Establish one clear owner for message routing, recipient resolution, event normalization, heartbeat/liveness tracking, and projection fan-out.

This is the most important architectural cleanup phase because later phases depend on having one event model and one delivery model. Do not treat this as a mechanical merge. Treat it as a boundary redesign.

### Primary Targets

- `Ichor.MessageRouter`
- `Ichor.Gateway.Router`
- `Ichor.Gateway.Router.EventIngest`
- `Ichor.Gateway.Envelope`
- `Ichor.Gateway.IntentMapper`
- `Ichor.Gateway.EventBridge`
- `Ichor.ProtocolTracker`
- `Ichor.Gateway.HeartbeatManager`
- relevant liveness/projection behavior in `Ichor.AgentWatchdog`

### Target Shape

The target architecture from the audit is:

- `Ichor.Messages.Bus`
- `Ichor.Events.Runtime`

Those names are less important than the boundary rules:

- one public message delivery API
- one normalized event stream
- one place that resolves routing targets
- one place that owns session aliases and liveness state
- zero duplicate bridges that reshape the same raw events in different ways

### Instructions

1. Define the canonical event model before moving code.
   - Write down the normalized event structure the system will use internally.
   - Include fields for session identity, source, target, event type, timestamps, transport metadata, and any audit metadata that downstream consumers require.
   - Identify the minimum event contract needed by memories ingestion, runtime dashboards, and project monitoring.
   - Do not merge implementations until this contract is explicit.

2. Choose the single public routing API.
   - Decide whether the surviving API is closer to `MessageRouter` or `Gateway.Router`.
   - Keep the simpler external call shape.
   - Standardize all callers on one function family for:
   - resolving recipients
   - sending messages
   - broadcasting to pattern targets such as `team:*`, `role:*`, `session:*`, and `agent:*`
   - recording audit/protocol side effects

3. Move recipient resolution into one owner.
   - Extract the target parsing and recipient lookup rules into the message bus or a tightly scoped private helper beneath it.
   - Remove duplicated resolution logic from any secondary router, event ingest path, or protocol tracking code.
   - Preserve behavior for all currently supported target patterns before deleting old code.

4. Fold event ingest into the normalized event runtime.
   - Replace `Gateway.Router.EventIngest` with a direct path into the normalized event runtime.
   - Raw sources should publish once into the runtime.
   - Every downstream sink should subscribe to normalized events, not rebuild identity and routing state independently.

5. Unify heartbeat and liveness ownership.
   - Compare `HeartbeatManager` and `AgentWatchdog` responsibilities carefully.
   - Keep one owner for:
   - heartbeat freshness
   - stale-agent detection
   - session/agent liveness projection
   - eviction or timeout decisions
   - If `AgentWatchdog` still needs pane scan policy or tmux-specific probing, keep that narrow concern there and emit facts into the unified event runtime.

6. Remove duplicate observability pipelines.
   - Replace `ProtocolTracker` and `EventBridge` only after the normalized event runtime can satisfy their consumers.
   - If a dashboard or persistence sink needs a specialized view, derive it from normalized events instead of keeping a second event-processing service.
   - Avoid introducing “temporary” dual-pipeline code that becomes permanent.

7. Migrate downstream consumers one by one.
   - Move memories ingestion first if it only needs normalized event facts.
   - Move runtime/control views next.
   - Move project-runtime subscriptions after the normalized stream is stable.
   - Delete the old bridge/tracker modules only after all consumers are converted.

8. Preserve auditability during the migration.
   - Add targeted tests around routing target resolution and event normalization.
   - Verify that audit logs, protocol traces, or equivalent telemetry still expose the information needed to debug delivery behavior.

### Constraints

- Do not keep two routing systems alive after the phase ends.
- Do not keep two heartbeat owners alive after the phase ends.
- Do not change message target semantics unless a mismatch is already a bug and the fix is explicitly documented.

### Deliverables

- One message bus for all delivery paths.
- One event runtime for normalized events and liveness.
- Removed gateway/router/event-tracker duplication.
- Downstream consumers depending on normalized events instead of custom reshaping.

### Validation

- Add or update tests for target resolution, message delivery, broadcast semantics, and liveness transitions.
- Exercise any existing integration tests covering tmux delivery, session routing, and event ingestion.
- Run `mix precommit`.

### Exit Criteria

- There is one authoritative path from raw event to normalized event to subscriber.
- There is one authoritative path from logical target to resolved recipients to transport delivery.
- `ProtocolTracker`, `EventBridge`, `HeartbeatManager`, and the extra router layer are removed or reduced to genuine boundary-specific code.

## Phase 3: Unify Run Lifecycle

### Purpose

Collapse the repeated runner implementations into one generic runner with mode-specific policy, then consolidate the project runtime around that shared lifecycle.

This phase should delete a large amount of repeated OTP code without weakening correctness around tmux lifecycle, completion, cleanup, and signal handling.

### Primary Targets

- `Ichor.Projects.BuildRunner`
- `Ichor.Projects.PlanRunner`
- `Ichor.Projects.RunProcess`
- `Ichor.Projects.RunnerRegistry` if still present
- `Ichor.Projects.ModeSpawner`
- `Ichor.Projects.Spawner`
- `Ichor.Projects.TeamLifecycle`
- `Ichor.Projects.TeamCleanup`
- `Ichor.Projects.Catalog`
- `Ichor.Projects.Discovery`
- `Ichor.Projects.DagAnalysis`
- `Ichor.Projects.Actions`
- `Ichor.Projects.HealthChecker`
- `Ichor.Projects.HealthReport`
- `Ichor.Projects.TeamSpecBuilder`
- `Ichor.Projects.GenesisTeamSpecBuilder`
- `Ichor.Projects.DagTeamSpecBuilder`

### Target Shape

The target architecture from the audit is:

- `Ichor.Projects.Runtime`
- `Ichor.Projects.Runner`
- `Ichor.Projects.Spawn`
- `Ichor.Projects.TeamSpec`

The important rule is that different run kinds should be represented as configuration or callbacks, not separate process stacks.

### Instructions

1. Document the common runner lifecycle before refactoring.
   - For `BuildRunner`, `PlanRunner`, and `RunProcess`, map the full lifecycle:
   - registration
   - startup
   - tmux launch or attachment
   - signal subscription
   - health checks
   - completion detection
   - teardown
   - emitted events
   - Keep this as a comparison table. The goal is to isolate true differences from naming differences.

2. Design a single runner state struct.
   - The unified runner should explicitly hold:
   - run identity
   - mode/config
   - team/session identifiers
   - supervisor and registry references
   - completion policy
   - periodic check policy
   - deadline/timeout policy
   - cleanup policy
   - Avoid storing mode-specific ad hoc fields unless they are namespaced under a config field.

3. Convert lifecycle differences into policy modules or config.
   - Express MES, Genesis, DAG, or any other run variant as data.
   - Use callbacks only where the behavior is truly different, such as completion predicates or health checks that require different external facts.
   - Do not create one callback module per historical runner unless the logic is materially different.

4. Introduce `Projects.Runner` while keeping old callers temporarily compatible.
   - Start by making one existing run type use the new runner end to end.
   - Once one path is stable, migrate the remaining run types to the new runner.
   - Remove the old runner modules after all paths are converted.

5. Collapse spawning into one boundary.
   - Merge `Spawner` and `ModeSpawner` into `Projects.Spawn`.
   - `Projects.Spawn` should accept a run request plus a mode descriptor and return the child spec or start result needed by the unified runner.
   - Keep all run creation and startup orchestration in this one boundary.

6. Collapse team-spec builders into one configurable builder.
   - Merge the specialized team spec builders into `Projects.TeamSpec`.
   - Support per-mode prompt/template generation through data or small callbacks, not separate builders with mostly shared structure.
   - Ensure the final team spec builder is the only place that decides how teams are described for a run.

7. Fold secondary project orchestration helpers into clear owners.
   - Move discovery, cataloging, DAG analysis, actions, and health reporting into `Projects.Runtime`, `Projects.Spawn`, `Projects.TeamSpec`, or one `Projects.Health` helper if that remains justified.
   - Delete any module that only exists because orchestration was over-sliced.
   - Keep a separate module only if it owns a genuinely reusable, complex concern.

8. Standardize cleanup.
   - There should be one cleanup path for tmux teams and related runtime artifacts.
   - Remove the split between generic cleanup, run-specific cleanup, and lifecycle wrappers.
   - Make cleanup behavior visible in the unified runner lifecycle.

9. Standardize emitted events and status reporting.
   - All run modes should emit the same lifecycle events where possible.
   - Run status transitions should be driven by the unified runner, not by duplicated mode-specific conventions.

### Constraints

- Do not regress run completion detection or tmux cleanup reliability.
- Do not preserve separate runners because of naming preference alone.
- Do not let mode-specific policy leak back into separate supervisor stacks.

### Deliverables

- One generic project runner.
- One spawning boundary.
- One team-spec builder.
- Substantially fewer project orchestration modules.

### Validation

- Add or update tests for startup, registration, signal handling, completion detection, timeout handling, and cleanup across multiple run modes.
- Run focused tests for project runtime behavior and any integration tests that cover tmux-backed teams.
- Run `mix precommit`.

### Exit Criteria

- `BuildRunner`, `PlanRunner`, and `RunProcess` are removed.
- Starting a run no longer requires separate mode-specific process implementations.
- Project orchestration code is centered on a small set of modules with clearly owned responsibilities.

## Phase 4: Collapse the Tool Surface

### Purpose

Reduce the implementation module count of the Ash/MCP tool surface while preserving external tool names and caller compatibility.

This phase is about internal consolidation, not externally visible churn. The audit is explicit that the tool names can remain stable even if the implementation collapses from dozens of modules to a handful.

### Primary Targets

- `lib/ichor/tools.ex`
- `lib/ichor/tools/agent/*.ex`
- `lib/ichor/tools/archon/*.ex`
- any lingering utility-only tool helpers after Phase 1

### Target Shape

The suggested grouping from the audit is:

- `Tools.AgentMessaging`
- `Tools.AgentMemory`
- `Tools.AgentExecution`
- `Tools.ArchonOps`
- `Tools.Genesis` if that surface must remain separate

The exact names can differ, but the grouping must follow real backend boundaries rather than prose categories.

### Instructions

1. Inventory the current tool surface by backend responsibility.
   - For each tool, record:
   - external MCP/Ash name
   - current resource module
   - runtime/resource boundary it calls
   - parameter normalization performed
   - output shaping performed
   - Use this inventory to group tools by actual backend ownership.

2. Freeze the external contract.
   - Identify every external tool name that must remain stable.
   - Preserve input and output shapes unless a mismatch is already a bug.
   - If any external change is unavoidable, document it explicitly and treat it as a compatibility event, not a side effect of refactoring.

3. Design the reduced resource layout.
   - Group tools by backend boundary, not by narrow verb category.
   - Avoid one resource per tiny concept such as “send”, “broadcast”, “lookup”, “team”, or “session” if they hit the same underlying runtime.
   - The goal is 3 to 5 resource modules, not 10 slightly larger micro-modules.

4. Move shared mapping and validation next to the new grouped resources.
   - Use private functions inside the grouped resources for argument normalization and result shaping.
   - Extract a shared presenter/helper only if multiple grouped resources genuinely need the same logic and the helper represents a real concept.
   - Do not rebuild `MapUtils` under a different name.

5. Update the tool registry/domain module.
   - Change `Ichor.Tools` so it registers the reduced set of implementation modules while preserving the external surface.
   - Verify that the manual listing of resources and tools still exposes the same tool names to callers.

6. Migrate tools incrementally by group.
   - Move one backend group at a time.
   - After each group, delete the old resource modules that are now empty.
   - Keep the system compiling between groups.

7. Align the new grouped tools with the simplified runtime APIs from earlier phases.
   - Agent messaging tools should call the unified message bus.
   - Agent execution/run tools should call the unified project/runtime boundaries.
   - Memory tools should call the surviving memory/event boundaries directly.

### Constraints

- External tool names should remain stable unless explicitly documented otherwise.
- Do not preserve an implementation module purely because the old file existed.
- Do not introduce new cross-cutting utility modules unless repeated code proves the need.

### Deliverables

- A drastically smaller number of tool/resource modules.
- Stable external MCP/Ash tool names.
- Cleaner alignment between tool groups and backend boundaries.

### Validation

- Add or update tests around tool invocation shape, especially argument parsing and returned map structure.
- Verify every externally exposed tool still resolves and executes correctly.
- Run `mix precommit`.

### Exit Criteria

- The tool implementation surface is reduced to a small set of grouped resources.
- `Ichor.Tools` remains stable for callers.
- Tool modules now mirror backend ownership instead of historical category naming.

## Phase 5: Clean Up Memory Store and Remaining Projections

### Purpose

Finish the simplification by reducing over-sliced memory-store helpers and removing any remaining duplicated projections that earlier phases exposed but did not eliminate.

This phase is last because it should be cleanup after the event and runtime boundaries are already stable.

### Primary Targets

- `Ichor.MemoryStore`
- `Ichor.MemoryStore.Blocks`
- `Ichor.MemoryStore.Recall`
- `Ichor.MemoryStore.Archival`
- `Ichor.MemoryStore.Persistence`
- `Ichor.MemoryStore.Tables`
- any remaining projection helpers that still rebuild event/session/liveness state outside the unified event runtime

### Target Shape

The audit recommends keeping:

- `Ichor.MemoryStore`
- `Ichor.MemoryStore.Storage`
- `Ichor.MemoryStore.Persistence`

The broader rule is to keep the memory boundary small and cohesive, not heavily decomposed around every operation.

### Instructions

1. Re-map memory-store responsibilities after Phase 2.
   - Re-check which responsibilities still belong to memory storage once normalized event handling is in place.
   - Remove any helper that existed mainly because event ingestion or projection state was previously fragmented.

2. Collapse helper modules by real responsibility.
   - Move table definitions and limits into the owning module unless multiple active modules truly need them.
   - Combine operational helpers such as block/recall/archival logic into a smaller `Storage` boundary if they are manipulating the same state with different verbs.
   - Keep `Persistence` separate only if disk/load-save concerns remain a real independent boundary.

3. Remove lingering duplicate projections.
   - Search for any module that still computes recent events, liveness, session aliases, or topology independently from the unified event runtime.
   - Either subscribe to normalized events or inline the remaining projection into the true owner.
   - This step should catch leftover cleanup from the earlier phases.

4. Standardize memory ingestion interfaces.
   - Ensure consumers send one normalized shape into memory ingestion.
   - Avoid special-case ingestion APIs for different event sources if the payloads are now equivalent.

5. Reduce config and constants sprawl.
   - Consolidate table names, caps, thresholds, and retention policies into the few modules that own them.
   - Avoid helper modules whose only purpose is to expose constants.

### Constraints

- Do not change retention or persistence behavior accidentally while merging modules.
- Do not create a new generalized abstraction if two modules can simply be merged.
- Do not leave a second event/projection pipeline alive because it seems harmless.

### Deliverables

- A smaller memory-store implementation.
- One clear ingestion path from normalized events into memory persistence/storage.
- Remaining projection duplication removed.

### Validation

- Add or update tests around memory write, recall, archival, persistence, and restart/load behavior as needed.
- Run `mix precommit`.

### Exit Criteria

- The memory-store boundary is reduced to a small number of cohesive modules.
- No major projection logic remains outside the unified event runtime unless it clearly belongs to a presentation boundary.
- The simplification effort ends with real structural reduction, not just renaming.

## Cross-Phase Rules

These rules apply to every phase.

### 1. Prefer ownership over indirection

If a module does not own state, policy, or a stable boundary, it is a merge candidate.

### 2. Normalize once

Event shape, routing target resolution, liveness state, and run lifecycle should each have one authoritative implementation.

### 3. Keep behavior stable while simplifying structure

Most of the work here is internal consolidation. Preserve public behavior unless an existing behavior is already a defect and the fix is deliberate.

### 4. Delete aggressively after migration

Do not leave “temporary” compatibility modules around after callers are migrated. That only recreates the same abstraction surplus under new names.

### 5. Validate every phase fully before continuing

Every phase should end with:

- focused tests for the changed area
- `mix compile`
- `mix precommit`

### 6. Judge success by boundary count, not file movement

The point is to end with a smaller number of real owners:

- a fleet core
- a projects runtime plus generic runner
- one event runtime
- one message bus
- a smaller memory boundary
- a drastically smaller tool surface

If a phase shuffles code without reducing duplicated ownership, it has missed the goal.
